# AI PM Coach - 架构说明

## 架构对比

### 旧架构：工具模式

```
用户 → /boundary → Skill → 输出结果
                ↓
            用户被动接收
```

问题：
- 一键生成，用户只是"使用者"
- AI 替用户做决策
- 用户不知道为什么这样设计

---

### 新架构：教练模式（Sub Agent）

```
用户 ←→ Sub Agent (AI PM Coach)
           ↓
       内部调用多个 Skill
           ↓
       引导用户做决策
```

优势：
- 引导式对话，用户是"学习者"
- 用户做决策，AI 只是引导
- 用户理解为什么这样设计

---


## 对话流程

```
用户：我想做一个 AI 客服助手
     ↓
教练：很好！让我引导你一步步思考...
     ↓
教练：首先，要解决什么核心问题？
     ↓
用户：客服响应太慢
     ↓
教练：明白了。这个问题影响有多大？
     ↓
用户：70% 用户抱怨过
     ↓
教练：那确实是大问题。我们评估一下 AI 是否适合？
     ↓
[内部调用 pain-check-skill]
     ↓
教练：根据你的描述，AI 适用性评分 22/25，适合用 AI。
      但你确认一下，你觉得呢？
     ↓
用户：确认
     ↓
教练：好的，那我们来看看 AI 能做什么、不能做什么？
     ↓
[内部调用 boundary-skill]
     ↓
...（继续引导式对话）
```

---

## 决策节点

在以下节点，Sub Agent 暂停，让用户做决策：

| 节点 | 决策内容 | 方式 |
|------|----------|------|
| 痛点确认 | 是否值得用 AI 解决？ | 展示评估结果，让用户选择 |
| 边界确认 | AI 能做什么、不能做什么？ | 列出边界，让用户补充/修改 |
| 阈值设置 | 置信度设多少？ | 提供选项，让用户选择 |
| 容忍度定义 | 什么错误可以接受？ | 解释风险，让用户决策 |
| 责任划分 | AI 出错谁负责？ | 列出选项，让用户选择 |

## 与旧版对比

| 项目 | ai-pm-prd-suite (v2.0) | ai-pm-coach (v1.0) |
|------|------------------------|-------------------|
| 架构 | 单个 Skill | Sub Agent |
| 交互方式 | 命令-输出 | 对话-引导 |
| 用户角色 | 使用者 | 学习者 |
| 决策者 | AI | 用户 |
| 输出时机 | 每次命令后 | 对话结束时 |
| 内部 Skills | 无 | 5 个封装 Skill |

## 适用场景

### ai-pm-prd-suite 适合：
- 用户清楚知道自己要什么
- 需要快速输出文档
- 单点问题咨询

### ai-pm-coach 适合：
- 用户不确定如何设计 AI 产品
- 需要学习和引导
- 完整的产品设计流程

## 技术实现

### Sub Agent 配置

```yaml
# SKILL.md
metadata:
  hermes:
    runtime: subagent  # 标识为 Sub Agent
```

### 内部 Skill 调用

Sub Agent 根据对话进展，内部调用不同 Skill：

```
对话阶段一 → 调用 pain-check-skill
对话阶段二 → 调用 boundary-skill
对话阶段三 → 调用 confidence-skill
对话阶段四 → 调用 hallucination-skill
对话阶段五 → 调用 prd-skill
```

### 用户不感知内部 Skill

用户只和 Sub Agent 对话，不需要知道内部有多个 Skill。
